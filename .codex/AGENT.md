# AGENT運用ルール（claude-send-todo）

## 誤認した理由（再発防止のための記録）

- ユーザー意図の主軸は「Skill運用」だったが、計画中の「契約を固定する」という文言を過大解釈し、実装手段をスクリプト化へ寄せた。
- `SKILL.md` の整備だけで満たせる段階にもかかわらず、「実行可能性の担保」を優先しすぎて、不要な実装を提案した。
- 空の `send-slack` skill を見た時点で、まずSkill定義を埋めるべきだったが、実装方式の確認前に拡張方針へ進んだ。

## このプロジェクトで行うこと

- Skill中心で運用する。
- まず `SKILL.md` の品質を上げて、手順・入出力・制約を明文化する。
- TODOは `TODO/YYYY/MM/DD/TODO.md` で管理する。
- 報告文は `create-report`（または `create-hokoku` / `create-nippo`）で生成する。
- Slack送信は `/send-slack` でのみ行う。
- 送信前に必ずプレビューし、`send` または `yes` の明示承認がある場合のみ送る。
- 単一リポジトリ・固定チャンネルをMVPの前提として維持する。

## このプロジェクトで行わないこと

- ユーザー要求がない段階で、Skill運用をスクリプト実装へ置き換えない。
- 自動送信しない（承認なし送信をしない）。
- 複数リポジトリ横断、動的な送信先推定、自動優先度決定をMVPで導入しない。
- 既存運用を壊す大規模変更を、合意なしで先行実装しない。

## 実装判断のゲート

- 追加実装（スクリプト、CLI、自動化）に進む条件:
  - Skill定義だけでは運用上の問題が再現的に発生している。
  - その問題が具体的に記録されている。
  - ユーザーから実装の明示依頼がある。
- 上記を満たさない場合は、Skill修正を優先する。
